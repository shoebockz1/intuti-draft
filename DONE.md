- **Done** — Draft state persistence via Upstash Redis (free tier). The live draft, transaction log, and pre-reset snapshots now survive server restarts/redeploys/crashes instead of living only in process memory — every mutation fires an async best-effort save, and the server rehydrates from Redis at startup before accepting requests. Deliberately optional (falls back to the old in-memory-only behavior with a warning if the env vars aren't set) and exposed on `GET /api/health` as `persistenceEnabled`. Verified twice: once locally with a full process kill, and again — the real test — on live production, by starting a real draft, making a pick, and using Render's actual "Restart service" action; the pick, transaction log, and pre-reset snapshot all survived intact on the real infrastructure, not just a local simulation. Also confirmed a hard reset correctly persists as wiped, and deployed the required Upstash env vars to Render.
- **Done** — Added a "5th Place Jump Pick" panel to the bottom of the left sidebar, after discovering (and proving with a real test) that the board itself can silently hide the jump pick's player: the jump pick and the 5th-place winner's own natural turn can share the same round, and the board only shows one pick per (round, owner) cell — the natural pick always wins that slot, hiding the jump's player entirely even though the underlying data stays correct. The panel is always correct regardless of the board's display limitation. Verified in a real browser against a live test draft, not just a build check. Originally showed a wall-clock timestamp for "when the pick was made," but that wasn't what was wanted — corrected to show where in the draft it happened instead (e.g. "After Round 1, Pick 2"), using the round/slot the jump pick was inserted at rather than the transaction log's timestamp. Verified end-to-end via a real test draft (two unprotected picks to trigger the jump, then the jump pick itself), correctly reading "After Round 1, Pick 2."
- **Done** — Auto-incrementing build version display (`v{commit count} ({short SHA})`), derived entirely from git/GitHub at build time via `scripts/generate-version.mjs` — no manual version-bumping step to ever forget, since it's physically tied to the actual commit being built. Wired into predev/prebuild/pretypecheck hooks in both client and server, shown as a small corner badge on the page and in `GET /api/health`. Replaces the ad-hoc bundle-hash-comparison trick used to verify deploys so far. Took two follow-up fixes after the first deploy to actually work: Render's build checkout turned out to be shallow (only 1 commit visible, so the count showed "v1" for everyone), and — once diagnosed via Render's real build log, not just local simulation — its shallow checkout has no git remote configured at all, so the first fix ("git fetch --unshallow") silently did nothing there even though it worked in a local shallow-clone test. Final fix reads the commit count from GitHub's public REST API instead (the "last" page number on the commits endpoint, with per_page=1, equals the exact count) when the local checkout is shallow; normal local dev is unaffected and still resolves entirely offline via `git rev-list`. Verified against a real shallow clone with its remote stripped (reproducing Render's exact condition), and confirmed live on the real hosted site post-deploy — `GET /api/health` correctly read `v41 (902dde8)`, matching the real commit count and exact deployed SHA.
- **Done** — Namespaced the Redis persistence key by environment (`intuti:draft-state` in production, `intuti:draft-state:dev` everywhere else). Found the hard way: local dev's `.env` was configured with the same Upstash credentials as production (only one free-tier database exists), so a local test session was silently writing to the exact same key production reads from — a local hard-reset done to clean up test data wiped a real live owner draft in progress. Confirmed via production's own snapshot history, which contained the local test draft's data, not the real one. Now dev traffic physically cannot touch production's key even if credentials are ever shared again. Verified live: local dev starts fully isolated (reports no draft in progress even while production has a real one live), and a full local start/reset round-trip left production's state completely untouched.
